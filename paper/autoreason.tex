\section*{Abstract}

We introduce autoreason, a method for iterative LLM refinement in subjective domains where no objective metric exists. Building on the autoresearch paradigm, autoreason addresses fundamental limitations of LLMs when iterating on subjective work: sycophancy, overcriticism, overcompromise, authorship bias, scope drift, and context collapse. Our approach uses fresh isolated agents per iteration, blind evaluation panels with ranked choice voting and Borda count aggregation, and conservative tiebreaking. In experiments on go-to-market strategy generation, autoreason achieved unanimous first-place preference (7/7 judges, 35/35 Borda) in a 5-way comparison against four baseline methods. We additionally ran autoreason on the writing of this paper itself, demonstrating the method's applicability to technical writing with ground-truth fact-checking.

\section{Introduction}

Large language models exhibit systematic failures when iterating on subjective work. Given a document and instructions to improve it, LLMs demonstrate sycophancy (strengthening existing arguments regardless of merit), overcriticism (finding flaws where none exist), overcompromise (attempting to satisfy contradictory feedback), authorship bias (preferring their own outputs), scope drift (expanding beyond original intent), and context collapse (losing track of quality through extended interaction).

These failures persist despite sophisticated prompting strategies. Recent work demonstrates that even in objective domains like code generation, iterative refinement with critique leads to degradation rather than improvement~\cite{slopcodebench}. The problem is more acute in subjective domains where no ground truth exists to anchor evaluation.

We present autoreason, extending Karpathy's autoresearch paradigm~\cite{autoresearch} to domains requiring subjective judgment rather than objective metrics. By treating each iteration as a fresh competition between three framings---conservative (A), adversarial (B), and synthetic (AB)---with blind evaluation, autoreason constructs a subjective fitness function through the evaluation process itself.

\section{Method}

Autoreason implements a three-way competition each iteration:

\textbf{Critic Phase}: A fresh LLM agent receives the current best document (A) and task description, then identifies problems. Each critic is a new API call with no conversation history from previous passes. The critic finds problems only---it does not suggest fixes.

\textbf{Response Phase}: Two separate fresh agents produce alternatives:
\begin{itemize}
\item \textbf{B (Adversarial revision)}: Receives the task, incumbent A, and the critique. Revises A to address valid criticisms.
\item \textbf{AB (Synthesis)}: Receives the task and both A and B with randomized labels. Keeps the strongest elements of each.
\end{itemize}

\textbf{Judge Phase}: A panel of 3 fresh judge agents performs blind evaluation. Each judge receives all three documents (A, B, AB) in randomized order with randomized labels. Judges rank the documents from best to worst. Rankings are aggregated using Borda count (3 points for first, 2 for second, 1 for third). Conservative tiebreak: the incumbent A wins ties.

Every role is a fresh, isolated agent with no shared context. The artifact is the thread of continuity, not the agent's memory.

The system converges when the incumbent A wins 2 consecutive passes. The original task prompt serves as the anchor throughout---judges evaluate ``which version best accomplishes the task as described,'' preventing drift toward complexity for its own sake.

\section{Experiments}

We evaluated autoreason on go-to-market strategy generation for an open-source Kubernetes CLI tool (5k GitHub stars, no revenue, 3-person team). All experiments used claude-sonnet-4-20250514 (temp=0.8 authors, 0.3 judges).

\subsection{Main Experiment}

We ran 26 passes ($\sim$160 LLM calls, $\sim$65 minutes). The system achieved 2 consecutive incumbent wins at passes 14--15 and 24--25 but never reached the original threshold of 3. Winner distribution: A 31\% (8/26), B 19\% (5/26), AB 50\% (13/26).

\subsection{5-Way Baseline Comparison}

We compared autoreason's pass 15 output against four baselines, each starting from the same initial output (847 words) with 15 passes. A 7-judge blind panel ranked all five outputs.

The conservative baseline---which barely changed the initial output---outperformed all three iterative approaches. Critique-and-revise, the method most commonly used for iterative refinement, scored lowest.

\subsection{Convergence and Quality Trajectory}

Pass 15 (first convergence point) defeated pass 25 (second convergence) 6--1 in a separate 7-judge blind panel. Ten additional passes degraded quality. The first convergence point is the quality ceiling.

\subsection{Baseline Visibility}

Judges comparing autoreason vs critique-and-revise: with the original output shown as baseline, 7--0 unanimous for autoreason. Without baseline, 3--2. Judges need a reference point to detect drift.

\subsection{Qualitative Improvement}

The initial output was a generic startup playbook: \$49/user pricing, \$100K MRR target with a 3-person team. After 14 passes, the output contained quantified pain (\$15K/incident $\times$ 6/yr), team pricing (\$1,499/mo), customer validation (50+ interviews, 75\% pilot success), competitive positioning against named tools, and unit economics (CAC \$2K, LTV \$54K). The adversarial process killed the unrealistic revenue assumptions.

\subsection{Prior Experiments}

The v2 architecture was informed by $\sim$1,800 LLM calls of earlier experiments that exposed: severe positional bias in non-blind evaluation (eliminated by randomized labels), single-judge noise (fixed by 3-judge panel), and shared-context contamination where the same agent rubber-stamped its own revisions (fixed by fresh isolated agents).

\section{Writing This Paper with Autoreason}

We ran autoreason on the writing of this paper itself, providing a second experimental domain and a test of the method's applicability to technical writing.

\subsection{Setup}

Claude Opus 4 for all roles (author, critic, judges). 3-judge panel, all Opus. The task prompt specified the paper's required sections, constraints, and---critically---the actual experimental numbers to use. All agents (critic, Author B, synthesizer) received the full experimental data as ground truth context.

\subsection{Ground-Truth Critic}

The key architectural addition: the critic received access to the actual experimental data (README, RESULTS.md, experiment JSONs) and was instructed to fact-check every claim. Without this, the initial Opus generation hallucinated an ablation study, fabricated confidence intervals, reported wrong model names, and got the A/B/AB role descriptions backwards. With ground-truth access, the critic caught these on pass 1 (B won, indicating real problems were found and fixed).

\subsection{Results}

The paper converged in 9 passes (54 Opus calls, $\sim$40 minutes):

\begin{center}
\small
AB $\to$ A $\to$ AB $\to$ AB $\to$ AB $\to$ AB $\to$ AB $\to$ A $\to$ A \checkmark
\end{center}

Five consecutive AB wins (passes 3--7) drove quality improvement. A then won passes 8--9 unanimously (scores 9-3-6 and 8-5-5), converging. When A wins, it wins decisively.

\subsection{Judge Panel Integrity}

An earlier run with a mixed-model panel (Opus + Sonnet + Gemini 2.5 Pro via OpenRouter) failed to converge after 11 passes. Investigation revealed Gemini's ranking format (\texttt{\#\#\# RANKING: [3], [1], [2]}) failed our parser, reducing the effective panel to 2 judges. With only 2 valid judges, Borda scores compressed and ties became frequent. Once fixed to 3 Opus judges, the incumbent from that run converged in 2 additional passes. This demonstrates that judge panel integrity is a prerequisite for convergence---a broken judge doesn't just add noise, it prevents the system from reaching stable equilibrium.

\section{Related Work}

\textbf{Autoresearch}~\cite{autoresearch} uses val\_bpb as an objective fitness function for autonomous experiments. Autoreason extends this to domains without objective metrics. \textbf{SlopCodeBench}~\cite{slopcodebench} shows agents produce progressively worse code each iteration---prompting doesn't fix it, confirming iterative degradation is structural. \textbf{ACE}~\cite{ace} documents context collapse in iterative rewriting. \textbf{LLM Council}~\cite{llmcouncil} uses multi-model panels for single-turn evaluation; autoreason adds iteration with role separation.

\section{Discussion}

\textbf{Bloat/prune oscillation.} On tasks with ambiguous scope, AB adds complexity while B prunes it. Word counts oscillated 847 $\to$ 1800 $\to$ 1644 $\to$ 1758 over 26 passes. This signals the task is underdetermined along the scope dimension---a useful finding rather than a failure.

\textbf{Conservative beats naive iteration.} The most striking baseline result: doing almost nothing (conservative, 862 words) outperformed 15 passes of ``improve this,'' ``harsh critic,'' and ``critique and revise.'' Iterating without structural protection is worse than not iterating.

\textbf{Ground-truth context prevents hallucination.} When writing this paper, the initial generation fabricated results. Giving the critic (and authors) access to actual experimental data eliminated hallucination by pass 1. This suggests a general pattern: the author writes from understanding, the critic verifies against evidence---mirroring how peer review works in practice.

\textbf{Beyond subjective domains.} Autoreason may prove valuable even where metrics exist but are insufficient. Code can pass tests while degrading in readability and maintainability~\cite{slopcodebench}. Autoreason could provide a complementary quality signal alongside objective metrics.

\textbf{Limitations.} Single task domain for the main experiment. No statistical significance testing with only 7 judges. No ablation studies isolating each design choice. All results from Anthropic models---generalization to other providers untested.

\section{Conclusion}

Autoreason demonstrates that systematic isolation and blind evaluation can construct effective fitness functions for subjective domains. The method achieved unanimous preference over four baselines and successfully wrote a version of this paper through 9 passes of self-refinement with ground-truth fact-checking. The key insight: the primary obstacle to LLM self-improvement in subjective domains is not capability but evaluation methodology.
